# AI 优先文档规范

## 目标

将所有产品文档（产品合同、设计规范、PRD、用户故事）从"人类 PM 阅读和执行的散文"转变为"AI 可无歧义解析和执行的接口定义"。人类审核确认，AI 负责执行。

## 当前状态

- ✅ office-hours 产出产品合同（Goals/Scope/AC/Risks）
- ✅ prd 产出 PRD 文档
- ✅ story 产出用户故事
- ❌ 文档格式以人类阅读为第一目标，AI 可执行性未被显式设计
- ❌ AC 描述中存在大量隐式假设（"应该很快""体验要好"）
- ❌ 没有歧义检查环节

## 方案设计

### 核心原则

1. **每条 AC 可被独立验证**：AC 描述中不能出现"等等""类似功能""同上"等引用性描述
2. **禁止模糊形容词**："快""好""流畅""直观" → 必须有量化标准
3. **四路径覆盖**：每个涉及用户交互的 AC 必须覆盖 happy/nil/empty/error 四条路径
4. **设计决策显式化**：不能有"默认""常规"等依赖 AI 常识的判断（AI 的"常识"不可靠）

### 文档改版：产品合同

当前 office-hours 产出格式：
```markdown
## Goals
- 用户能快速创建和管理项目
- 仪表盘展示关键指标

## Scope
- 项目 CRUD
- 仪表盘：MRR、Churn、NPS

## AC
- 用户点击"新建项目"后能看到创建表单
- 仪表盘数据实时更新

## Risks
- 数据量大时仪表盘可能加载慢
```

改为 AI 可执行格式：
```markdown
## Goals（纯方向，不直接驱动实现）
- 用户能在 3 步内完成项目创建（选模板 → 填名称 → 确认）
- 仪表盘在 5 秒内展示 4 项关键指标的当前值和 7 天趋势

## Scope（精确边界）
- 项目 CRUD：创建、查看列表、编辑名称/描述、归档（非删除）
  - 不包含：项目模板管理、项目权限分配、项目内子任务
- 仪表盘：MRR、Churn Rate、NPS Score、DAU
  - 不包含：自定义指标、导出、同比环比

## AC（每条独立可测）
### AC-01: 项目创建
- Given: 用户在项目列表页
- When: 点击"新建项目"按钮
- Then:
  [happy] 弹出创建表单（模板下拉 + 名称输入 + 确认按钮），提交后跳转到项目详情
  [nil]   模板列表为空时，显示"暂无可用模板，将创建空白项目"
  [empty] 用户不填名称点确认 → 名称字段标红"项目名称不能为空"
  [error] 网络异常提交失败 → toast "创建失败，请重试"，表单内容保留不丢失

### AC-02: 仪表盘加载
- Given: 用户进入仪表盘页面
- When: 页面加载数据
- Then:
  [happy] 5 秒内展示 4 个指标卡片（含当前值 + 7 天趋势小图）
  [nil]   某个指标无数据 → 卡片显示"--"，hover 提示"暂无数据"
  [empty] 全新项目无任何数据 → 显示引导文案 + "接入数据源"按钮
  [error] 接口超时 → 卡片显示"加载失败"，提供"重试"按钮

## Risks（含量化标准）
- 仪表盘数据量 > 10K 条时查询可能 > 5 秒
  → 缓解：分页加载 + 数据库索引 + 缓存层
  → 验收：p95 延迟 < 5 秒（100 并发下）
```

### AC 歧义检查清单

每份产品合同定稿前，AI 自动运行以下检查（作为 office-hours 的最后一个步骤）：

```
AC 歧义检查清单：
[ ] 每条 AC 是否包含 Given/When/Then 三要素？
[ ] Then 是否覆盖了 happy/nil/empty/error 全部四条路径？
[ ] 是否存在模糊形容词（快/好/流畅/直观/合理/合适/足够）？
[ ] 是否存在引用性描述（"同上""类似""与 XX 相同"）？
[ ] 每条 AC 是否可以独立编写测试而不需要额外的上下文推断？
[ ] 所有提到的 UI 元素是否有具体位置和交互描述？
[ ] 数字是否有单位（ms / s / 条 / 个 / %）？
[ ] 错误状态是否有用户可见的处理方式？

通过率要求：8/8 项全部通过才标记为"AI-Ready"。
```

### 对下游 skill 的影响

```
office-hours:
  产出 → 满足歧义检查清单的产品合同
  新增 → "AI 可执行性检查"作为最后一个步骤

prd:
  输入 → AI-Ready 产品合同
  新增 → 检查合同歧义后再撰写（如合同不合格，先退回 office-hours）

story:
  输入 → AC 列表
  新增 → 每条 story 必须映射到至少 1 条 AC，标注四路径覆盖情况

feature-breakdown:
  输入 → AI-Ready AC 列表
  新增 → 拆解时检查每个 task 是否关联到明确的 AC
```

### 渐进式落地策略

不需要所有旧文档一次性改版：

```
Phase 1: 新文档强制 AI-Ready
  - office-hours 新增产出后，自动运行歧义检查
  - 不通过 → 不允许标记"定稿"

Phase 2: PRD / story 适配
  - 读取合同后先检查歧义标记
  - 合同未 AI-Ready → 提醒用户先修合同

Phase 3: 存量文档渐进升级
  - 每次打开旧合同时，提示"此合同未经过 AI-Ready 检查，要现在升级吗？"
```

## 风险与防御

| 风险 | 概率 | 防御 |
|------|------|------|
| 过度规范化导致用户厌烦 | 中 | 只在新文档上强制，存量文档渐进；提供"跳过检查"选项 |
| AI 生成的 AC 流于形式（填满四路径但内容空洞） | 中 | plan-review 管道审查内容质量，检查清单只管格式 |
| 简单功能不需要四路径 | 低 | 纯内部函数/非交互功能可标注 N/A |
