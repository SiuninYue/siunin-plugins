# MVP 类型与设计

## MVP 是什么

**MVP = 验证核心假设的最小产品**

MVP 不是：
- ❌ 功能不全的 v0.1
- ❌ Bug 一堆的半成品
- ❌ 「先上了再说」的借口

MVP 是：
- ✅ 只做验证假设必需的功能
- ✅ 能完整解决一个核心问题
- ✅ 能收集到有效反馈数据

---

## MVP 类型矩阵

### 按验证目的分类

| MVP 类型 | 验证什么 | 成本 | 周期 |
|---------|---------|-----|------|
| Landing Page | 需求存在吗？ | 极低 | 1-3天 |
| 视频演示 | 产品概念有吸引力吗？ | 低 | 1周 |
| 众筹预售 | 用户愿意付钱吗？ | 低 | 2-4周 |
| 人工模拟 | 流程可行吗？ | 中 | 2-4周 |
| 单一功能 | 核心价值成立吗？ | 中 | 4-8周 |
| Concierge | 服务流程对吗？ | 中 | 2-4周 |

---

## MVP 类型详解

### 1. Landing Page MVP

**适用场景**：验证需求是否存在

**做法**：
1. 用 Carrd/Notion/Webflow 做一个落地页
2. 描述产品价值和功能
3. 收集用户邮箱/预约
4. 看转化率判断需求

**模板**：
```markdown
## Landing Page 设计

**页面结构**：
1. 头条：一句话说清楚产品价值
2. 痛点：用户遇到什么问题
3. 方案：你怎么解决
4. 功能：核心功能列表
5. CTA：留下邮箱/预约演示

**流量来源**：
- [来源1]：预计 X 人
- [来源2]：预计 X 人

**成功标准**：
- 页面转化率 > X%
- 收集 X 个邮箱
```

**经典案例**：Buffer 的第一个版本就是一个 Landing Page

---

### 2. 视频演示 MVP

**适用场景**：验证产品概念是否有吸引力

**做法**：
1. 制作演示视频（可以是动画/录屏）
2. 展示产品如何工作
3. 收集反馈和预约

**模板**：
```markdown
## 视频演示计划

**视频时长**：2-3 分钟

**内容结构**：
1. 问题描述 (30s)
2. 解决方案展示 (60s)
3. 核心功能演示 (60s)
4. CTA (30s)

**发布渠道**：
- [渠道1]
- [渠道2]

**成功标准**：
- 观看完成率 > X%
- 预约/留资 > X 人
```

**经典案例**：Dropbox 在产品做出来之前，发布了一个演示视频，收获了 7 万预约

---

### 3. 众筹预售 MVP

**适用场景**：验证用户是否愿意付钱

**做法**：
1. 在众筹平台（Kickstarter/Indiegogo/摩点）发起项目
2. 设置不同的奖励档位
3. 看预售数据判断市场

**模板**：
```markdown
## 众筹预售计划

**平台选择**：[平台名]

**目标金额**：¥X

**档位设置**：
| 档位 | 价格 | 回报 | 预计人数 |
|-----|------|------|---------|
| 早鸟 | ¥X | [回报] | X 人 |
| 标准 | ¥X | [回报] | X 人 |
| 豪华 | ¥X | [回报] | X 人 |

**成功标准**：
- 达成率 > 100%
- 平均客单价 > ¥X
```

**优点**：真金白银，比问卷靠谱

---

### 4. 人工模拟 MVP (Wizard of Oz)

**适用场景**：验证流程是否可行，还没有自动化能力

**做法**：
1. 用户看到的是产品界面
2. 背后是人工处理
3. 验证流程后再开发系统

**模板**：
```markdown
## 人工模拟计划

**用户看到的**：
[描述前台体验]

**实际处理方式**：
[描述后台人工处理流程]

**人力成本**：
- 需要 X 人
- 每单处理时间 X 分钟

**扩展瓶颈**：
- 当用户量达到 X 时，需要自动化

**成功标准**：
- 用户满意度 > X
- 流程验证可行
```

**经典案例**：Zappos 早期没有库存，收到订单后创始人亲自去商场买鞋再发货

---

### 5. 单一功能 MVP

**适用场景**：验证核心价值是否成立

**做法**：
1. 只做一个核心功能
2. 做到极致
3. 验证用户是否买单

**模板**：
```markdown
## 单一功能 MVP

**核心功能**：[功能描述]

**为什么是它**：
[这个功能最能代表产品核心价值的原因]

**不做的功能**：
- [功能A]：因为 [原因]
- [功能B]：因为 [原因]

**最小可用标准**：
- [ ] [标准1]
- [ ] [标准2]
- [ ] [标准3]

**成功标准**：
- 用户使用率 > X%
- 留存率 > X%
```

---

### 6. Concierge MVP

**适用场景**：验证服务型产品的流程

**做法**：
1. 人工为用户提供服务
2. 观察用户需求和痛点
3. 逐步总结标准化流程

**模板**：
```markdown
## Concierge MVP 计划

**服务内容**：
[描述要提供的服务]

**第一批用户**：
- 数量：X 人
- 来源：[渠道]
- 筛选条件：[条件]

**服务流程**：
1. [步骤1]
2. [步骤2]
3. [步骤3]

**收集的数据**：
- [数据1]
- [数据2]

**成功标准**：
- 用户愿意付费
- 流程可标准化
```

## MVP 检查清单

### 设计前检查

- [ ] **明确要验证的假设**：这次 MVP 要验证什么？
- [ ] **定义成功标准**：什么数据说明成功？
- [ ] **选择合适的类型**：哪种 MVP 最适合验证这个假设？
- [ ] **评估时间成本**：能在 2 周内做出来吗？

### 设计中检查

- [ ] **只解决一个问题**：不贪多
- [ ] **保留核心体验**：不因为是 MVP 就降低核心体验质量
- [ ] **能收集反馈**：有数据埋点/反馈渠道
- [ ] **可以快速迭代**：不是做完就扔

### 上线后检查

- [ ] **数据是否符合预期**：达到成功标准了吗？
- [ ] **用户反馈是什么**：定性反馈如何？
- [ ] **下一步是什么**：继续/转型/放弃？

## MVP 常见错误

| 错误 | 问题 | 正确做法 |
|-----|------|---------|
| **功能太多** | 不知道哪个功能生效 | 只做核心功能 |
| **体验太差** | 用户流失不是因为不需要 | 核心体验要好 |
| **没有标准** | 不知道怎么判断成功 | 提前定义成功标准 |
| **不收集反馈** | 只有数据没有洞察 | 同时收集定量+定性 |
| **做完就不管** | 错过迭代机会 | 持续观察和迭代 |

## MVP 到产品的路径

```
       MVP
        │
    ┌───┴───┐
    │       │
  验证     验证
  失败     成功
    │       │
    ↓       ↓
  转型    迭代
  or      or
  放弃    规模化
```

### 验证成功后

1. **识别核心价值**：用户真正喜欢什么
2. **补充必要功能**：不是锦上添花，是必须有
3. **提升体验质量**：从能用到好用
4. **扩大用户规模**：规模化获客
