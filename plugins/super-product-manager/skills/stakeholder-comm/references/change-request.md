# 需求变更沟通

## 变更沟通原则

| 原则 | 说明 |
|-----|------|
| **影响透明** | 清楚说明对时间、资源、范围的影响 |
| **给出选项** | 不只是提问题，要给解决方案 |
| **有截止日期** | 明确决策时间点 |
| **书面记录** | 口头同意不算数 |

## 变更控制流程

```
发现变更需求
    │
    ▼
评估影响
    │
    ▼
提交变更请求
    │
    ▼
相关方评审
    │
    ├─ 批准 → 更新计划 → 执行
    │
    ├─ 拒绝 → 维持原计划
    │
    └─ 需要更多信息 → 补充分析 → 重新评审
```

## 变更沟通技巧

### 如何说服决策者

| 策略 | 做法 |
|-----|------|
| **量化影响** | 用数字说话，不要用形容词 |
| **对比方案** | 至少给 2 个选项 |
| **关联业务目标** | 说明和业务目标的关系 |
| **风险透明** | 不隐瞒风险 |
| **给出建议** | 明确推荐哪个方案 |

### 应对拒绝

| 情况 | 回应 |
|-----|------|
| 预算不够 | 是否可以分期？是否可以缩减范围？ |
| 时间不够 | 是否可以延期？是否可以分批上线？ |
| 不认可必要性 | 补充数据支撑，说明不做的后果 |
| 需要更多信息 | 明确需要什么信息，约定时间反馈 |

## 常见错误

| 错误 | 问题 | 正确做法 |
|-----|------|---------|
| **只说要改** | 没说影响 | 量化影响 |
| **只有一个方案** | 没有选择 | 至少 2 个方案 |
| **口头同意** | 事后扯皮 | 书面记录 |
| **没有截止日期** | 决策拖延 | 设定决策时间 |
| **等出问题再说** | 太晚了 | 提前预警 |
