# idea-concretization

## 延迟加载规则

- 默认只使用本文件内容
- 只有在用户明确要求细节或需要模板时，才加载 references 中的附件
- 未加载附件时，先给摘要与下一步

## 核心说明

这是一个“单入口”想法技能，自动分两种模式：

- **创意生成模式**：用户没有明确想法时，从线索生成候选方向
- **想法改进模式**：用户已有想法时，进行具体化与改进建议

本技能不做市场认可与证据判定（交由 `skills/market-validation/SKILL.md`）。

## 核心流程

1. 判断模式（生成 vs 改进）
2. 明确想法一句话定义
3. 判断想法类型（决定追加问题）
4. 补全用户与场景
5. 梳理现有替代与差异化
6. 输出可验证假设清单

## 输出结构（概览）

- 模式判断结果
- 想法一句话定义
- 目标用户与场景
- 现有替代方案与差异化
- 核心假设列表
- 需要补充的信息
- 改进建议（功能/呈现/定位）

## 示例对话与输出

### 创意生成模式示例
**用户输入**: "我想做一个帮助远程团队协作的工具，但没具体想法"
**技能输出**:
- 模式判断: 创意生成模式（无明确想法）
- 生成方向:
  1. 实时协作白板工具（亮点: 直观可视化；风险: 竞品多）
  2. 异步视频沟通平台（亮点: 解决时区问题；风险: 技术复杂）
  3. 团队健康度监测工具（亮点: 数据驱动；风险: 隐私敏感）
- 推荐进入改进模式: 实时协作白板工具

### 想法改进模式示例
**用户输入**: "我想做一个健身打卡小程序，帮助上班族坚持运动"
**技能输出**:
- 模式判断: 想法改进模式（有明确想法）
- 想法类型: 应用类（App/Web）
- 一句话定义: 为上班族解决健身难坚持问题，通过小程序打卡社区，与Keep的差异是更轻量、社交驱动
- 追加问题: 核心场景与频次？当前替代工具？关键价值能否量化？
- 核心假设: 用户真的有健身坚持问题；该场景高频；差异化成立

## 使用边界

- 本技能不做市场验证结论
- 证据与实验设计交由 `skills/market-validation/SKILL.md`

## 模式判定规则

- 用户没有明确想法，只给领域/兴趣/痛点 → 加载 `ideation.md`（创意生成）
- 用户已给出具体想法 → 进入想法改进模式

## 判定示例（内部规则）

触发创意生成的输入示例：
- “我想做点新东西，但没想法”
- “给我几个方向/点子”
- “围绕X领域给一些灵感”

触发想法改进的输入示例：
- “我想做一个XX工具/产品”
- “我的想法是……”
- “这是我的方案/功能描述”

## 基线信息（必须提供）

1. 用户是谁（具体到生活/工作状态）
2. 解决什么问题（发生频率与痛苦程度）
3. 现有替代方案是什么
4. 资源与约束（时间/预算/渠道）

## 想法类型与按需加载规则

先识别想法类型，并按规则加载对应附件：

- 应用类（App/Web）→ `app.md`
- 消费品/食品 → `consumer.md`
- SaaS/B2B → `saas.md`
- 服务/咨询 → `service.md`
- 平台/双边 → `platform.md`
- 硬件/IoT → `hardware.md`
- 内容策划（产品导向）→ `content.md`
- 教育/培训产品 → `education.md`
- 游戏/娱乐产品 → `game.md`
- 社交产品 → `social.md`
- 工具类产品 → `tool.md`

仅加载匹配类型的附件；不加载其他类型，避免无关信息与 token 浪费。

## 类型识别指南

### 常见想法类型映射
- **软件/数字化产品**：App/Web、SaaS、工具类
- **实体产品**：消费品/食品、硬件/IoT
- **服务/平台**：服务/咨询、平台/双边、教育/培训
- **内容/创意**：内容策划、游戏/娱乐、社交产品

### 识别决策提示
1. **交付形式**：软件、硬件、服务、内容？
2. **用户关系**：B2C、B2B、双边市场？
3. **价值载体**：功能、实物、知识、体验、连接？
4. **收入模式**：销售、订阅、广告、交易佣金？

### 不确定时的处理
- 如果想法涉及多个类型，选择最核心的类型
- 如果不确定，可以描述想法的关键特征进行辅助识别
- 创意生成模式会涵盖多种可能类型

**关键假设（按验证优先级排序）**
1. [假设1：用户真的有该问题]（最高优先级）
2. [假设2：该场景高频/高痛]
3. [假设3：差异化成立]
4. [假设4：关键约束可解决]

- **P0（必须验证）**: 用户问题真实性、场景价值
- **P1（重要验证）**: 差异化可行性、关键约束
- **P2（后续验证）**: 风险控制、商业模式

### 需要补充的信息

- [缺失信息1]
- [缺失信息2]
- [追加问题的缺失项]

### 建议产出文档（按需）

**通用文档（适用于大多数项目）**
1. **PRD**：明确范围、边界、验收标准，避免开发返工
2. **用户故事**：把需求拆成可执行任务与验收场景
3. **竞品分析**：验证差异化是否成立
4. **路线图**：把阶段目标与资源配置可视化

**类型特定文档**
- 各想法类型有特定的文档建议，详见对应模板文件
- 依据"想法类型"加载对应模板获取类型特定文档建议

### 相关技能（下一步行动）

完成想法具体化后，可根据需要转入以下相关技能：

**核心验证技能**
- **market-validation**：市场验证 - 设计验证实验，收集市场证据
- **user-interview**：用户研究 - 深入访谈目标用户，验证需求假设

**文档与规划技能**
- **prd**：产品需求文档 - 编写详细的产品需求规格
- **user-story**：用户故事 - 拆解需求为可执行开发任务
- **roadmap**：产品路线图 - 规划阶段目标和资源配置
- **tech-spec**：技术规格 - 设计系统架构和技术方案

**分析与决策技能**
- **prioritization**：优先级排序 - 对功能/任务进行优先级决策
- **finance-basics**：财务分析 - 评估成本、收益和投资回报
- **legal-compliance**：合规检查 - 检查产品法律合规要求

**使用建议**
- 市场验证建议在想法具体化后立即进行
- 文档编写可在验证关键假设后进行
- 根据项目阶段和资源选择最相关的技能
