# office-hours

## 延迟加载规则

- 默认只使用本文件内容
- 本 skill 无附件，所有方法论已内联

## 核心说明

前置产品澄清技能（planner 角色）。输出仅含目标、范围、验收标准、风险，严禁写入任何技术实现路径。

---

## 需求澄清三板斧

在落笔产品合同之前，必须覆盖三个维度：

**项目上下文**：
- 这个产品解决什么问题？为谁解决？
- 我们按多大规模设计？（用户量、吞吐量、数据量）
- 与现有系统的关系是什么？（替代/补充/集成）

**技术约束**：
- 性能硬指标（吞吐、延迟上限）
- 数据一致性要求（强一致/最终一致）
- 安全与合规硬性要求
- 团队现有技术栈和运维能力

**资源边界**：
- 团队规模和能力画像
- 交付时间线（硬死线 vs 大致目标）
- 长期维护的预期（谁来维护、多久）

## 粒度控制原则

产品合同中每条目标和范围声明应该：
- 可在一个迭代（1-2 周）内独立验证
- 有清晰可测的完成标准（"用户能做什么"而非"系统有什么"）
- 不绑定具体技术实现路径
- 按依赖链排序：基础能力 → 核心逻辑 → 外部接口 → 用户界面 → 集成打磨

**太粗**（需要拆解）：
- "构建用户增长系统"（方向太模糊）
- "优化全站性能"（无法验证完成）

**太细**（实现细节，不属于产品合同）：
- "添加 Redis 缓存层"
- "选择 React 还是 Vue"

## 依赖识别

明确标注每个目标的前置条件：
- A 必须在 B 之前落定（硬依赖）
- A 未定则 B 范围无法锁死（软依赖）
- 对软依赖，写出假设条件："假设 A 已就绪，则 B 范围如下"

## 验收标准定义

每条验收标准必须是可执行、可通过产品行为验证的：

**好的验收标准**：
- "用户首次打开应用时，3 秒内看到引导页"
- "订单提交后，用户收到确认邮件且订单状态为 pending"

**差的验收标准**：
- "体验要好"（无法客观验证）
- "系统要稳定"（没有量化门槛）

## 产品合同落盘结构

遵循 4 节强制结构落盘到 `docs/product-contracts/`：

1. `## Goals` — 要达成的产品目标（用户视角）
2. `## Scope` — 明确边界（包含什么，明确排除什么）
3. `## Acceptance Criteria` — 可测试的验收条件（每条 1-2 句）
4. `## Risks` — 关键风险和已知未知（不含 mitigations，那是技术层的事）

## 递进原则

1. **Progressive Disclosure**：从简单版本开始，逐层补细节
2. **YAGNI**：不为假设的未来需求扩大当前合同范围
3. **Pragmatism**：最好的产品合同是能指导开发团队开始行动的合同，而非最完美的合同

## 防惰性机制

### 反谄媚规则

| 禁用表述 | 必需替代 |
|---------|---------|
| "这个想法很好" | "这个想法的致命假设是…" |
| "可以考虑一下" | "具体建议：…，因为…" |
| "这个范围很清晰" | "以下边界仍模糊，需要锁定：…" |

### Pushback Patterns（用户回避时的犀利追问）

| 用户行为 | 追问 |
|---------|------|
| 给出模糊目标（"做一个好的体验"） | "请给出一个用户场景：谁在什么情况下做了什么，结果是什么" |
| 跳过技术约束（"先不管这个"） | "如果延迟超过 Xms 或数据不一致，产品层面的失败表现是什么？" |
| 验收标准无法量化 | "如何在不看代码的情况下验证这条标准通过了？" |
| 范围持续膨胀 | "这条新增范围解决了哪个已命名用户的哪个已命名痛点？" |
| 跳过风险识别 | "如果三个月后这个假设被证伪，合同中哪条 AC 会失效？" |

### 产出前自我检查（落盘前必须确认）

- [ ] 4 节结构（Goals/Scope/AC/Risks）全部存在且非空
- [ ] 每条 AC 可在产品行为层面独立验证（无需读代码）
- [ ] 合同正文中无技术实现路径（无框架名、无架构词）
- [ ] 依赖关系已标注（硬依赖/软依赖/假设）
- [ ] 风险条目具体（非"可能出现问题"而是"假设 X 失败则 Y 范围无法交付"）

## 子 Agent 审查指令

产出 artifact 并同步 PROG 后，使用 Agent 工具 spawn 一个独立审查 agent。

给它的 prompt：
> 读取 `<artifact_path>`，从 completeness/consistency/clarity/scope/feasibility 5 个维度审查，返回 JSON：
> `{"pass": bool, "issues": [{"dimension": "...", "severity": "blocking|advisory", "description": "..."}]}`

收到结果后评估：
- `pass: true` → 流程结束
- 仅有 `advisory` issues → 记录到 PROG update details，不修改 artifact，流程结束
- 有 `blocking` issues → 修改 artifact → 追加 `sync_planning_update(category="status")` update → 重新 spawn 审查（最多 2 次迭代）
- 2 次迭代耗尽仍有 blocking → 追加 `category=risk` update，向用户报告并建议人工介入
