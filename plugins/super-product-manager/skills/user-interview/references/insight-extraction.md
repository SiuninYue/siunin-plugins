## 洞察提取方法

### 亲和图法 (Affinity Mapping)

**步骤**：
1. 把每个发现写在便签上
2. 把相似的发现归到一起
3. 给每组起一个主题名称
4. 识别主要模式

**示例**：
```
┌──────────────────────────────────────────────────────┐
│  主题：现有工具太复杂                                  │
├──────────────────────────────────────────────────────┤
│  - P1："功能太多了，我只用 10%"                        │
│  - P3："学习成本太高，培训了一周"                       │
│  - P5："每次找功能都要搜索"                            │
│  - P7："新员工上手很慢"                               │
└──────────────────────────────────────────────────────┘
```

### 频率分析

| 发现 | 提及人数 | 比例 | 优先级 |
|-----|---------|-----|-------|
| 现有工具太复杂 | 7/10 | 70% | 高 |
| 价格太贵 | 5/10 | 50% | 中 |
| 缺少某功能 | 3/10 | 30% | 低 |

## 从发现到洞察

### 发现 vs 洞察

| 发现 (Finding) | 洞察 (Insight) |
|---------------|---------------|
| 描述现象 | 解释原因 |
| "5/10 人说工具太复杂" | "用户需要的不是更多功能，而是更少决策" |
| 表面观察 | 深层理解 |

### 洞察生成公式

```
发现 + 原因 + 机会 = 洞察
```

**示例**：
- **发现**：70% 的用户只用产品 10% 的功能
- **原因**：他们不知道其他功能存在，也不想学
- **机会**：简化界面，把核心功能放最显眼
- **洞察**：用户不需要"强大的工具"，需要"傻瓜式的工具"

### 洞察质量检查

好洞察应该：
- [ ] **有数据支撑**：不是拍脑袋想的
- [ ] **有指导意义**：知道该怎么行动
- [ ] **够具体**：不是"用户想要更好的体验"
- [ ] **够意外**：不是常识

## 用户原话记录技巧

### 为什么要记原话

| 你的总结 | 用户原话 |
|---------|---------|
| "用户觉得产品复杂" | "每次打开软件我都想关掉它" |
| 信息丢失 | 保留情感和细节 |

### 记录原话的方法

1. **录音转文字**：用讯飞、飞书妙记等工具
2. **即时标记**：听到重要的话立刻记下
3. **事后回听**：重听录音补充遗漏

### 原话分类标签

| 标签 | 说明 | 示例 |
|-----|------|------|
| 💢 痛点 | 表达不满、挫折 | "每次都要重新输入，烦死了" |
| 💡 期望 | 表达希望、想法 | "要是能自动保存就好了" |
| 🎯 行为 | 描述具体行动 | "我一般会先导出成 Excel" |
| 💰 价值 | 表达愿意付出 | "这个功能值 100 块" |

## 常见分析错误

| 错误 | 症状 | 解决 |
|-----|------|------|
| **确认偏差** | 只看支持你观点的数据 | 主动找反例 |
| **样本太小** | 2-3 人就下结论 | 至少 5 人以上 |
| **过度推断** | 从个例推广到全体 | 注意频率和比例 |
| **忽视异常值** | 把不符合预期的数据忽略 | 异常值可能是金矿 |
| **只看说的** | 忽略用户的实际行为 | 行为比语言更真实 |
