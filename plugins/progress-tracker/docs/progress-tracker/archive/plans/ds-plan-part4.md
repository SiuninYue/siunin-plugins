# PROG全量规范化计划 - 第四部分：实施计划与测试策略

## 设计目标与原则

### 核心目标
一次性完成 `schema 2.1` 规范化、自动合约导入、结构门禁、生命周期审计、复盘分离、摘要加速，并修复关键缺口：`/prog-done` 后必须进入"集成与收尾门禁"，避免 main 长期不完整和 worktree 残留。

### 实施原则
1. **零侵入体验**：现有 `/prog-*` 命令用法不变；日常用户不新增学习负担；**对外公开的新增命令仅保留 2 个排障入口**，内部实现会有更多辅助命令。
2. **增量交付**：按阶段实施，每个阶段都有明确的验收标准和可观测指标。
3. **测试先行**：契约测试先红灯后转绿，确保实现符合设计规范。
4. **安全迁移**：确保数据完整性，提供回滚机制和降级开关。

## 集成策略与现有代码兼容性

阶段2先告警不阻断，阶段3再切换为阻断，降低迁移期摩擦。

## 命令清单与公开策略

### 对外公开命令（用户可见）
1. **`validate-feature-readiness --feature-id <id>`** - 检查功能就绪状态
2. **`set-lifecycle-state --feature-id <id> --state <target> --reason "<text>"`** - 设置生命周期状态

### 内部排障命令（流程驱动，不在主帮助展示）
1. **`prog-set-finish-state --feature-id <id> --status <status> [--reason "<text>"]`** - 设置收尾状态（由`/prog-done`二段式流程调用）
2. **`prog-cleanup-worktree <feature_id> [--force]`** - 清理worktree
3. **`prog-check-finish-gates [--verbose]`** - 检查收尾门禁状态
4. **`prog-fix-state --feature-id <id> [--auto]`** - 修复功能状态
5. **`prog-validate-feature --feature-id <id>`** - 验证功能完整性
6. **`prog-check-timeouts [--include-snoozed]`** - 检查超时状态
7. **`prog-resolve-timeout <feature_id>`** - 处理超时
8. **`prog-snooze-timeout <feature_id> [--days <n>]`** - 暂停超时提醒
9. **`prog-ignore-timeout <feature_id>`** - 标记超时为已查看
10. **`prog-rollback --to <timestamp>`** - 回滚到指定时间点

### 命令公开策略
- **零侵入体验**：日常用户只需使用现有`/prog-*`命令，无需学习新命令
- **排障入口**：仅公开2个排障命令供高级用户使用
- **流程驱动**：内部命令由特定流程自动调用（如`/prog-done`→`prog-set-finish-state`）
- **文档一致性**：主帮助只展示用户需要知道的命令，内部命令有独立文档

## 验收标准与质量门禁

### 阶段验收标准
每个实施阶段都有明确的DoD（Definition of Done），只有达到DoD才能进入下一阶段。

### 整体验收标准
1. **功能完整性**：所有设计功能完整实现
2. **性能达标**：大项目下`/prog`响应时间在可接受范围内
3. **测试覆盖率**：新增测试覆盖所有关键场景，全绿通过
4. **向后兼容**：现有用户工作流不受影响
5. **文档完整**：命令文档与实际实现一致

### 可观测指标
1. **迁移失败率** = `迁移失败次数 / 总迁移尝试`（目标：< 1%）
2. **自动修复成功率** = `自动修复成功次数 / 需要修复的总数`（目标：> 95%）
3. **收尾完整率** = `有明确收尾状态的done次数 / 总done次数`（目标：100%）
4. **门禁拦截率** = `正确拦截的非法操作 / 总非法操作尝试`（目标：100%）
5. **响应时间P95** = `/prog`命令的95百分位响应时间（目标：< 2秒）

## 风险控制与应急预案

### 已知风险
1. **迁移数据丢失风险**：通过备份机制和事务回滚控制
2. **并发写冲突风险**：通过文件锁和重试机制控制
3. **性能退化风险**：通过summary投影和缓存优化控制
4. **用户学习成本风险**：通过零侵入设计和渐进式门禁控制

### 应急预案
1. **数据损坏恢复**：`prog-rollback`命令回滚到指定时间点
2. **功能降级**：`PROG_DISABLE_V2=1`环境变量切换回v2.0逻辑
3. **紧急修复**：保留所有变更的审计记录，便于问题定位和修复

### 监控与告警
1. **每日自检任务**：扫描pending_finish、脏worktree、状态脑裂并自动出报告
2. **异常检测**：监控迁移失败率、自动修复成功率等关键指标
3. **用户反馈机制**：通过审计记录和updates链提供问题追溯能力

## 附录：关键设计决策

### 设计决策1：零侵入体验
**决策**：保持现有`/prog-*`命令用法不变，新增功能对日常用户透明。
**理由**：降低用户迁移成本，确保平滑过渡。
**影响**：内部实现复杂度增加，但用户体验保持稳定。

### 设计决策2：增量迁移策略
**决策**：采用阶段化实施，每个阶段都有明确验收标准。
**理由**：降低实施风险，便于问题定位和修复。
**影响**：实施周期延长，但质量更有保障。

### 设计决策3：硬阻断与软告警分层
**决策**：`/prog-next`前对未收尾功能硬阻断，其他问题软告警。
**理由**：确保关键问题被解决，同时不阻塞正常工作流。
**影响**：用户需要处理硬阻断，但体验更可靠。

### 设计决策4：审计与状态同事务
**决策**：所有状态变更和审计记录在同一事务内完成。
**理由**：确保审计链完整，便于问题追溯。
**影响**：事务复杂度增加，但数据一致性更有保障。

### 设计决策5：summary投影缓存
**决策**：引入`progress_summary.json`作为投影缓存。
**理由**：提升大项目下的性能体验。
**影响**：需要维护主状态与summary的一致性。
